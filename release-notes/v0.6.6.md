# pi-pretty 0.6.6

## Summary
This release fixes an extension load crash where the pi CLI (via jiti's CJS interop) couldn't resolve `@earendil-works/pi-ai/base` because pi-ai's `exports` map only has `"import"` conditions for subpath exports. The fix replaces CJS `require()` with ESM dynamic `import()` in the SDK fallback path.

## What changed
- **Fix extension load crash** — When `pi-pretty` is loaded as a pi extension, the fallback `require("@earendil-works/pi-coding-agent")` triggers jiti's CJS-to-ESM interop. Subpath exports like `@earendil-works/pi-ai/base` fail because pi-ai only defines `"import"` (not `"require"`) export conditions. CJS falls back to legacy resolution, looking for `dist/index.js/base` instead of `dist/base.js`.
- Fix: `piPrettyExtension()` is now `async` and uses `await import(...)` instead of `require(...)`. Dynamic `import()` uses ESM resolution, correctly resolving all subpath exports through the package `exports` map.
