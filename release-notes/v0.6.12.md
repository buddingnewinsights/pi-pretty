# pi-pretty 0.6.12

## Summary

Hardens published installs against the Pi resume crash reported on Pi `0.79.9` with `pi-pretty@0.6.11`.

## What changed

- Removed the nested runtime `@earendil-works/pi-coding-agent` dependency from the published extension install.
- Kept the Pi SDK as a dev-only dependency for types/builds.
- Made FFF commands/indexing independent from the optional SDK tool-factory import.
- Registered `/fff-health`, `/fff-rescan`, session FFF indexing, and FFF autocomplete before attempting the optional SDK import.
- Stopped destroying the native FFF finder during `session_shutdown`; Pi can emit shutdown/start during resume or session switching while the same process continues, and native teardown there is too risky.

## Why

`v0.6.10`/`v0.6.11` fixed missing dependencies by putting the full Pi SDK in the extension runtime dependency tree. That made published installs different from local extension runs: the extension loaded a nested Pi SDK inside the host Pi process. On Pi `0.79.9`, that can still crash during resume.

The safer extension shape is: do not ship/load a nested Pi host runtime. FFF should initialize without the SDK import. Pretty wrappers are now best-effort when the SDK factory import is available, but FFF health/indexing stays alive if it is not.
