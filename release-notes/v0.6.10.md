# pi-pretty 0.6.10

## Summary

This patch fixes published npm installs where pi-pretty could activate without registering the pretty tool wrappers or FFF commands/indexing.

## What changed

- Moved `@earendil-works/pi-coding-agent` into runtime `dependencies`.
- Bumped package metadata to `0.6.10`.
- Refreshed `package-lock.json`.

## Why

`dist/index.js` dynamically imports `@earendil-works/pi-coding-agent` during extension activation to construct the wrapped built-in tool definitions.

Local extension runs had this package available from `devDependencies`, so FFF and tool registration worked locally. Published Pi npm extension installs are isolated under Pi's npm extension root and only install runtime dependencies. Because the SDK package was only a dev dependency, the activation import failed in published installs and pi-pretty returned before registering:

- pretty `read` / `bash` / `ls` / `find` / `grep` / `multi_grep` tools
- `/fff-health`
- `/fff-rescan`
- FFF session indexing/status

## Verification

- `npm run typecheck` ✅
- `npm run build` ✅
- Clean `npm pack` install with `--omit=dev` ✅
- Clean activation registered tools: `read,bash,ls,find,grep,multi_grep` ✅
- Clean activation registered commands: `fff-health,fff-rescan` ✅
- Clean mocked `session_start` reported `fff=FFF indexed` ✅

## Upgrade notes

```bash
npm ls --prefix .pi/npm @heyhuynhgiabuu/pi-pretty @earendil-works/pi-coding-agent @ff-labs/fff-node typebox
```

Inside Pi, verify with:

```text
/fff-health
```
